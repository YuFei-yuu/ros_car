# 基于 YOLO 的巡逻异常检测任务：方案与可行性分析

## 1. 任务背景

实验室、工业车间和仓储区域中的日常巡检任务，通常不仅要求机器人按照固定路线移动，还要求机器人能感知环境中的人员、遗留物或异常目标。单纯的地图导航只能解决“到哪里去”的问题，无法判断“到达后看到了什么、是否需要处理”。

因此，本任务在已有小车建图、定位、导航、避障能力基础上，引入 YOLO 目标检测，将视觉识别结果接入巡逻状态机，使巡逻任务从“固定路径执行”扩展为“到点观察与安全处置”。

## 2. 任务描述

系统保留已有能力：

- SLAM建图；
- AMCL定位；
- Nav2路径规划与局部避障；
- 命名点导航；
- 零速度停车保护。

新增能力：

- YOLO目标检测；
- 多帧检测结果统计；
- 目标类别语义合并；
- 风险等级判断；
- 告警停车与人工 reset。

完整流程：

```text
启动导航与YOLO
→ RViz设置初始位姿
→ 发送 /yolo_patrol/start
→ 读取 yolo_patrol_course.yaml 中的巡检点
→ 导航到 inspect_1
→ 停车稳定
→ YOLO检测
→ 根据类别判断动作
→ 继续 inspect_2 / inspect_3 或等待人工确认
→ 完成后输出巡检摘要
```

## 3. 巡检点配置

建图完成后，需要提前设置巡检点位置。配置文件建议为：

```bash
/home/ubuntu/ros2_ws/src/yolo_patrol/config/yolo_patrol_course.yaml
```

每个巡检点需要记录：

```yaml
inspect_1:
  x: 1.0
  y: 0.0
  yaw_deg: 0.0
```

含义：

- `x`：地图坐标系中的横向位置；
- `y`：地图坐标系中的纵向位置；
- `yaw_deg`：小车到达后车头朝向角，影响摄像头能否看到目标。

如果测试场地或地图变化，只需要重新标定这些巡检点，不需要修改 YOLO 巡检状态机。

## 4. 为什么行驶中不立即响应目标

行驶中 YOLO 可能识别到目标，但系统默认不立即停车或改变导航任务，原因是：

- 运动中画面抖动，误检概率更高；
- 目标可能只是短暂进入画面；
- 直接打断 Nav2 导航会让状态机复杂；
- 容易出现频繁停车、取消任务和重新规划。

因此本任务采用：

```text
移动中轻量监听，到点后正式检测。
```

移动中检测结果只用于辅助判断。如果移动中看到候选目标，但到点后没有确认，系统会标记为 `missed_after_moving`，提示人员检查巡检点朝向、目标摆放和观察时间。

## 5. 当前测试策略

根据前期 YOLO 单项测试结果，第一版采用以下类别：

| 目标 | 识别类别 | 处理动作 |
|---|---|---|
| 人 | person | 安全停车，等待 reset |
| 竖立矿泉水瓶 | bottle | 记录后继续巡逻 |
| 手机或遥控器 | cell phone / remote | 告警停车，等待 reset |

其中，`cell phone` 和 `remote` 可能存在跳变，因此在任务层面合并为“疑似电子设备或遗留物”。

## 6. 人员现场操作策略

### 6.1 启动任务

```bash
ros2 topic pub --once /yolo_patrol/start std_msgs/msg/Bool "{data: true}"
```

### 6.2 人工确认后继续

用于 `person`、`cell phone`、`remote` 触发停车后：

```bash
ros2 topic pub --once /yolo_patrol/reset std_msgs/msg/Bool "{data: true}"
```

### 6.3 查看状态

```bash
ros2 topic echo /yolo_patrol/status
```

### 6.4 查看 YOLO 输出

```bash
ros2 topic echo /yolo/object_detect
```

### 6.5 强制停车

```bash
ros2 topic pub --once /controller/cmd_vel geometry_msgs/msg/Twist "{}"
ros2 topic pub --once /cmd_vel geometry_msgs/msg/Twist "{}"
```

## 7. 常见问题与处理

| 问题 | 可能原因 | 人员操作 |
|---|---|---|
| `no_target` | 目标不在画面中或置信度低 | 调整目标位置，必要时降低阈值 |
| `no_yolo_data` | YOLO或相机没有发布消息 | 检查 `/depth_cam/rgb0/image_raw` 和 `/yolo/object_detect` |
| `missed_after_moving` | 移动中看到目标，到点后未确认 | 调整 `yaw_deg` 或增加观察时间 |
| `safety_stop` | 检测到 person | 确认安全后发送 reset |
| `alert_stop` | 检测到 cell phone / remote | 记录结果后发送 reset |
| 导航失败 | 初始位姿、点位或地图问题 | 回到 `course_nav` 单点验证 |
| 小车低电量 | 供电不足导致节点异常 | 停止测试，充电后继续 |

## 8. 可行性分析

本方案可行的原因：

- 已有 Nav2 导航功能可复用；
- 已有 YOLO 节点可发布 `/yolo/object_detect`；
- 目标检测结果已有标准消息格式；
- 巡检点数量较少，便于标定；
- 第一版使用 COCO 模型即可完成闭环，不依赖立即训练新模型。

主要风险：

- 通用 YOLO 对横躺透明瓶识别不稳定；
- 手机可能在 `cell phone` 与 `remote` 之间跳变；
- 巡检点朝向不准会导致看不到目标；
- 小车电量不足会影响导航和相机。

对应策略：

- 使用语义合并，而不是强行依赖精确类别；
- 到点后多帧投票；
- 先用 `person / bottle / cell phone / remote` 跑通闭环；
- 后续再考虑自训练模型。

## 9. 后续改进方向

如果后续要提升识别稳定性，可以选择两条路线：

1. 针对要识别的物品重新采集数据、标注并训练 YOLO 模型；
2. 在现有 COCO 模型基础上调整参数，包括置信度阈值、观察时间和多帧确认次数。

建议优先完成系统闭环，再优化模型精度。
